# Docs Update Standard

Docs are part of the product and the operating system for the team. They are not optional cleanup after implementation.

## Update Rules

- Create or update the active spec before non-trivial work.
- Update product docs when user value, positioning, or scope changes.
- Update architecture docs when boundaries, data flow, storage, or contracts change.
- Update security docs and runbooks when risk posture, auth, storage, incident handling, or release controls change.
- Add or revise ADRs when a cross-cutting decision changes.

## Stable vs Task-Specific Knowledge

- Put stable reusable lessons in shared docs, rules, or skills.
- Put task-specific findings, tradeoffs, and open questions in the active spec.
- Reconcile the two after implementation so repeated work does not depend on memory.

## Completion Standard

Docs are current when a new engineer or agent can understand what was decided, why it was decided, how it was verified, and what remains an explicit open decision.
