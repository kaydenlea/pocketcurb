# Overview

State the web change and the intended business or user outcome in one or two paragraphs.

# User or Problem Context

Describe the audience, funnel stage, and trust or discovery problem being solved. Connect the work back to the real product thesis without importing SEO needs into mobile product decisions.

# Requirements

List functional, non-functional, content, analytics, privacy, and operational requirements. Separate confirmed claims from future ideas.

# Clarifying Questions

Record the questions that materially affect architecture, content truth, conversion strategy, privacy disclosures, or analytics design. If none remain, say so explicitly.

# File Plan

List the files to create or modify and the role of each file, including content, tracking, schema markup, shared packages, tests, and docs.

# Interfaces and Types

Describe the data contracts, content structures, API calls, waitlist payloads, and analytics events that will be introduced or changed.

# Edge Cases

Document unsupported-claim risk, attribution gaps, bot submissions, privacy disclosure issues, localization considerations, and failure states for waitlist or contact flows.

# Verification Plan

List required checks: lint, typecheck, unit, integration, end-to-end, visual verification, metadata validation, structured-data validation, analytics validation, and security review as applicable.

# Checklist

- planning complete
- product and content claims aligned with source docs
- implementation sliced
- verification run for each slice
- docs reconciled
- review complete
- release gate identified

# Design Decisions

Record the selected information architecture, messaging choices, measurement approach, and any rejected alternatives with rationale.

# Review Notes

Capture self-review, independent AI review, and human review. Include any content-risk or compliance findings.

# Final Reconciliation

Summarize what shipped, what changed from plan, what verification completed, what docs changed, and any explicit open decisions that remain.
